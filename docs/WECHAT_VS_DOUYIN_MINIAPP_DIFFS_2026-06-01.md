# 闪蜂速运 微信小程序与抖音小程序差异说明 2026-06-01

## 一句话结论

这套项目已经有一条现成路线：

- 微信版源码：`shanfeng-express-miniapp`
- 抖音版导出包：`shanfeng-express-douyin-miniapp`

不是从零重写，而是：

1. 保留业务结构
2. 替换平台 API
3. 替换登录
4. 替换支付
5. 补抖音审核规则和宿主兼容

## 当前代码里已经做掉的部分

`scripts/export-douyin.mjs` 现在已经会自动做这些事：

- `.wxml` 转 `.ttml`
- `.wxss` 转 `.ttss`
- `wx:` 转 `tt:`
- `wx.` 转 `tt.`
- `wechat-login` 改成 `douyin-login`
- 微信相关文案改成抖音相关文案
- 输出独立的抖音小程序目录

所以“作为抖音小程序需要改什么”，你现在不是从 0 开始，已经完成了一半以上的渠道转换。

## 真正还要改的 6 类东西

### 1. 登录体系

微信版：

- `wx.login`
- 服务端 `/api/v1/auth/wechat-login`

抖音版：

- `tt.login`
- 服务端 `/api/v1/auth/douyin-login`

当前项目这一层已经有基础。

### 2. 支付体系

微信版：

- 服务端 `JSAPI/小程序下单`
- 前端 `wx.requestPayment`

抖音版：

- 服务端先走抖音担保交易下单
- 前端 `tt.pay`

这个差异是最大的，不是简单把 `wx.requestPayment` 改成 `tt.pay` 就行。

### 3. 审核规则

抖音版比微信版更严格的点，你现在最该记住两条：

- 如果有支付场景，要接平台统一支付，不要引导到站外支付
- 首次发版时，手机号授权能力有平台限制，登录最好先走更轻的路径

### 4. 宿主兼容

抖音小程序不只跑在抖音，还可能跑在头条等宿主。

官方文档明确建议：

- 用 `tt.canIUse`
- 做基础库版本兼容
- 必要时判断 `tt.getSystemInfoSync().appName`

### 5. 平台配置

微信版常见配置：

- 合法 request 域名
- upload/download 域名
- 业务域名

抖音版常见配置：

- request 合法域名
- 抖音开放平台能力权限
- 支付能力申请
- 审核测试账号和说明

### 6. 文案与运营入口

抖音不是简单改 logo。

要一起检查：

- 联系方式文案
- 登录文案
- 客服入口
- 分享入口
- 是否存在站外引流表述

## 作为抖音小程序，最少必须改什么

如果只做可上传可联调版本，最少改这 8 项：

1. 用 `npm run export:douyin` 生成渠道包
2. 写入真实 `DOUYIN_APP_ID`
3. 把 API 地址切到真实后端
4. 服务端确保 `/api/v1/auth/douyin-login` 可用
5. 检查所有 `wx.` 是否清干净
6. 检查文案里是否还残留“微信”
7. 支付位改成抖音担保支付链路
8. 在抖音开发者工具真机跑完整下单链路

## 如果要做到可审核通过，还要额外注意

### 不要保留这些风险点

- 站外支付引导
- 外链支付文案
- 首发版本强依赖手机号授权
- 纯测试文案
- 只做壳页面、关键流程不可用

### 必须补这些能力

- 客服/售后入口
- 可用登录链路
- 可用支付链路
- 可用订单链路
- 审核说明和测试账号

## 结合你当前项目的最优解

最优解不是维护两套完全分裂代码，而是：

- 继续以 `shanfeng-express-miniapp` 作为主源码
- 用 `scripts/export-douyin.mjs` 生成抖音渠道包
- 把平台差异集中在：
  - 登录
  - 支付
  - 配置
  - 文案
  - 审核规则

这样后面维护成本最低。
